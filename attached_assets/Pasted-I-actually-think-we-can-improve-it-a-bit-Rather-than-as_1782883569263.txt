I actually think we can improve it a bit.

Rather than asking “Are you DeafBlind?” or “Do you use hearing aids?”, I’d have Faience begin by asking how the player experiences the world. That makes the setup respectful and future-proof. Someone might wear hearing aids one day, CI processors another, or use no auditory input at all.

I’d put this into Anchor Part 4.

# RESONANCE PROJECT ANCHOR
## Part 4 — Accessibility, Knowledge, AI Memory and Immediate Tasks
Accessibility First
Accessibility is the foundation of Resonance.
Every gameplay mechanic must be playable through multiple input and output methods.
No player should ever be locked out of the game because of disability.
Accessibility options are independent.
Players may combine any options they wish.
--------------------------------------------------

The player may change these choices at any time.
Faience learns preferences over time but never locks the player into a profile.
--------------------------------------------------
Accessible Output
If the player selects no auditory input, the game immediately:
Enables captions.
Removes action timers unless required by multiplayer.
Enables Braille display support if available.
Replaces spoken urgency with haptics, visual cues and tactile music patterns.
Important information must never depend on sound alone.
--------------------------------------------------
Accessible Input
Every important action must support multiple activation methods.
Examples include:

Emergency accessibility menu
No interaction should require a single mandatory gesture.
--------------------------------------------------
Orientation Layer
Reality always has priority over immersion.
Faience maintains two simultaneous models:
The real world.
The game world.
When navigation becomes important, Faience lowers game volume and announces real-world orientation.
Examples:
Turning left onto West Market Street.
Destination five blocks ahead.
Crosswalk approaching.
Construction ahead.
After the announcement, immersion fades naturally back in.
Players may replay the last orientation announcement instantly through any assigned input.

--------------------------------------------------
Multiplayer
Different professions cooperate.
Different knowledge matters.
Players succeed because they think differently.
Accessibility preferences remain individual.
One player's accessibility settings must never alter another player's experience.
--------------------------------------------------
Immediate Task
Do not brainstorm.
Initialize the GitHub repository.
Create the repository structure.
Create every Markdown document.
Populate them with the approved project design.
Treat the repository as the single source of truth.
Every future design decision becomes an edit to those documents rather than another conversation.

One thing I’d add after we get the repository running—and I don’t want to lose this idea—is an Accessibility Lab. That’s where we’d prototype things like Braille display support, tactile music, wearable haptics, smart canes, switch devices, and new interaction methods without cluttering the core game design. I think that would fit naturally under design_notes/ until ideas mature enough to become part of the main game.
