# Project Constitution

This document governs how the Resonance project is designed, documented, and decided. It has authority over all other documents. When another document conflicts with the constitution, the constitution wins until it is formally amended here.

## Purpose

Resonance is a science-fiction game franchise whose world is rendered primarily through sound and whose foundation is accessibility. This constitution exists so the project can grow without losing its identity, and so any contributor or assistant can understand the rules quickly.

## Single source of truth

The GitHub repository is the single source of truth. Every approved design decision belongs inside the Markdown documentation, not only in chat. A decision that lives only in a conversation is not yet real. To make it real, edit the relevant living document and record the change in the Decision Log.

## Core philosophy

These pillars guide every decision.

- Audio is gameplay.
- Story teaches mechanics.
- Accessibility is foundational.
- Perspective creates replayability.
- Knowledge is progression.
- The AI is a character.
- Technology exists to serve people.
- Curiosity is always rewarded.
- Every meaningful decision gains something while sacrificing something.
- Players should feel remembered.

## What the project explores

Resonance explores real themes through interactive gameplay rather than lectures. These themes include identity, adaptation, disability, history, trust, memory, privacy, survival, artificial intelligence, and ethics.

## Accessibility as foundation

Accessibility is the foundation of the universe, not a feature added at the end. No player may be locked out of the game because of disability. Every gameplay mechanic must be playable through multiple input and output methods. The Accessibility Bible defines the details, and no design may violate it.

## Documentation rules

- Every document uses proper Markdown headings, in a logical hierarchy.
- Every document must be navigable by screen readers.
- Never use bold text in place of a heading.
- Avoid unnecessary tables. Prefer headings and lists.
- Use descriptive link text rather than raw URLs where possible.
- Maintain one living document per file. Never create competing versions.
- GitHub provides the version history, so there is no need for versioned copies of a file.

## How decisions are made

1. A decision is proposed and discussed.
2. When approved, the relevant living document is edited to reflect it.
3. The change is summarized in the Decision Log with a date.
4. If the decision raises new uncertainties, they are added to Open Questions.

## Amending this constitution

This constitution may be amended by editing this document. Every amendment must also be recorded in the Decision Log so the reason and date are preserved.
