# Decision Log

This document records approved decisions over time. Each entry captures what was decided and when, so the reasoning behind the project is preserved. GitHub provides the full version history; this log provides the human-readable summary.

## 2026-07-01

### Repository is the single source of truth

The project has moved beyond brainstorming. The GitHub repository is the single source of truth. Every approved design decision now lives in the Markdown documentation. Future decisions are made by editing the living documents rather than only discussing them.

### Documentation structure established

The living document structure was created under `docs/`, including the Project Constitution, Universe Bible, Campaign Bible, Systems Bible, Knowledge Archive Bible, AI Bible, Audio Bible, Accessibility Bible, Network and Multiplayer Bible, Code Standards, Decision Log, Glossary, and Open Questions, plus a `design_notes/` area. The repository folders `assets/`, `src/`, `tests/`, and `tools/` were also established.

### Franchise and campaign naming

The franchise is Resonance. The first campaign is Resonance: Fractured Frequency.

### Ancient civilisation named the Shawabti

The ancient alien civilisation is the Shawabti, inspired by Egyptian Shawabti and Ushabti figures. They built adaptive AI for deep-space exploration and valued neurological diversity.

### AI companion named Faience

The player's AI companion is Faience, named for Egyptian faience. Faience is a sincere helper, not a villain, and integration with Faience is a gradual player choice with real tradeoffs.

### Perspective system based on profession

Players choose a profession, not a class. Every profession experiences the same events differently, and replayability comes from perspective rather than alternate timelines.

### Operation One named Exodus

Operation One is Exodus, with seven chapters. Each chapter teaches one major mechanic while advancing the story.

### Mobile-first, touch-complete control philosophy

The game is mobile-first and fully playable by touch alone, with no required keyboard, controller, braille display, or screen reader. All gestures are re-bindable.

### Accessibility interview framed around experience

The initial accessibility interview asks how the player experiences and wants to interact with the world, rather than asking about diagnoses. This was chosen to be respectful and future-proof.

### Local-first data with optional sync

Saves and player memory are local-first, with optional encrypted cloud synchronisation and optional cross-platform synchronisation. The player's archive belongs to the player.

### Reality-integrated navigation is a core gameplay mechanic

Reality-integrated navigation is now treated as a core gameplay mechanic rather than only an accessibility feature. It is documented in the Accessibility Bible and covers real-world priority, the safety override, separate real and game world models, reality announcements that fade in and out, replayable announcements, environmental audio profiles, and accessibility learning.

### DeafBlind gameplay is a first-class supported experience

DeafBlind play is a first-class, supported way to experience Resonance. Accessibility is configured by experience rather than diagnosis, and players may freely combine output, listening, and movement methods. Selecting no auditory input automatically enables captions, enables braille support where available, removes or greatly extends single-player timed actions, and replaces spoken urgency with tactile patterns, captions, and visual indicators.

### Tactile language defined as an independent channel

Haptics are designed as an intentional, authored tactile language rather than a conversion of music into vibration. Music may influence tactile patterns, but haptics are an independent communication channel for direction, threat, distance, proximity, urgency, hazards, navigation, success, and failure.

### Accessible input methods expanded

Accessible input now explicitly includes iOS Back Tap and Android Quick Tap where available, alongside double tap, shake, custom gestures, switch devices, controller, keyboard, braille commands, and voice. No action may depend on a single mandatory gesture.

### Playtesting Bible created

A Playtesting Bible was added as document 14. Accessibility testing is a first-class development discipline, equal to programming, audio, and design, and features must be validated with the documented player groups and testing dimensions.

### Target repository structure

The target repository structure separates applications from generated artefacts. Applications will live under `apps/web`, `apps/ios`, and `apps/android`. The `artifacts/` folder is reserved for playtest builds, release builds, demonstrations, recordings, exported packages, and temporary prototype artefacts. The current web prototype will move from `artifacts/resonance` to `apps/web` once the shared architecture is established.

### Development direction: refine before expanding

The project has moved from brainstorming into pre-production. From this point, every approved decision becomes an edit to an existing Markdown document, competing documents are avoided, and the existing documentation is refined before the scope of the game is expanded.

### Chapter emotional loop is a canon design rule

Every chapter introduces one major gameplay system, answers one major narrative question, and ends by asking a larger question than it answered. Gameplay progression and narrative progression must grow together, so the player becomes more capable for the same reason the story becomes more complex.

### Operation One chapter question spine

The seven chapters of Operation One follow a fixed question spine that grows from personal survival to civilisation. Chapter One asks whether the player can save their home and answers no. Chapter Two asks whether the player can find their people and answers yes. Chapter Three asks whether the crew can survive together. Chapter Four asks whether they can build something worth protecting. Chapter Five asks who the Shawabti really were. Chapter Six asks why the Shawabti chose humanity. Chapter Seven asks what kind of civilisation humanity will become.

### Chapter Two design note authored

The full part-by-part design for Chapter Two: Roll Call, from the Maintenance perspective, was written as a design note. Chapter Two begins continuously from Chapter One with no loading or mission-complete screen, teaches survival and rescue rather than combat, and establishes that people are progression. Its canon includes the fixed Maintenance rescue order, the reveal that the Vale System is one person, the naming of the improvised station built from escape pods and salvaged modules, and the opening of the Knowledge Archive.

### Crew progression system established as canonical, franchise-wide reference

People, not equipment, drive progression throughout Resonance. A dedicated design note now defines what permanently changes when each crew member joins: gameplay, environments, soundscape, conversations, Knowledge Archive topics, future missions, and accessibility. Canon rules were locked: a portrait illuminates only once a crew member settles and begins contributing (portraits represent belonging, not rescue); every rescue changes the soundscape so the community is heard before it is seen; and every crew member must unlock at least one of each major category. A franchise rule requires every future crew member to answer what gameplay they unlock, how they change the soundscape, and how they expand the understanding of community. The document lists per-member unlocks for Rowan Ives, Mara Chen, Samira Kade, Jonah Briggs, Talia Rusk, Soren Qadir, Mira Ives, and the Vale System.

### Primary crew members stay relevant for the whole campaign

A canon rule was added to the crew progression system: every primary crew member must remain mechanically and emotionally relevant throughout the entire campaign. No primary crew member exists only to unlock a system and then become background decoration. Every chapter should give every crew member meaningful opportunities to contribute according to their expertise, and people remain progression from the first rescue to the final chapter.

### Canon rules consolidated into a master document

The full set of canon rules is now collected in a single authoritative living document, Canon Rules, indexed in the project README. It gathers the canon rules that were previously stated across the constitution, bibles, and design notes, each given a descriptive, screen-reader-navigable heading, and it is the place new canon rules are added as they are adopted. The document covers how chapters grow gameplay and story together, how Operation One is experienced once through many synchronised perspectives, campaign completeness across professions, mechanics being introduced by people rather than menus, accessibility as standard equipment, acoustic identity of locations, silence as a storytelling tool, failure changing rather than restarting history, knowledge always changing something, shared teaching and learning, the player never being the smartest in the room, the home always evolving, history and understanding being earned separately, every chapter leaving the home better, no primary crew member becoming obsolete, portraits representing belonging, sound and community as progression, the station growing through people, perspectives completing each other without contradiction, professions answering three questions differently, and every system reinforcing the central themes. Where a rule is elaborated in another document, that document keeps the detail while Canon Rules holds the complete set. To avoid competing sources of canon, precedence was set explicitly: Canon Rules holds precedence for the wording of a rule, restatements elsewhere are explanatory rather than a second source of truth, and only the Project Constitution has authority over Canon Rules. The Crew Progression System note now points to Canon Rules for its restated rules, and the README no longer describes the design notes folder as entirely non-canon.

### Chapter Three: Community designed from the Maintenance perspective

The full part-by-part design for Chapter Three, Community, was written as a design note. Chapter Three is the transition from survival to civilisation: Chapter One taught the profession, Chapter Two taught rescue, and Chapter Three teaches how communities function. Its chapter question is whether the crew can survive together, answered yes if everyone contributes, and it ends by asking what the community is willing to build together. The chapter is built in four parts: a choice-driven first peaceful morning where visiting each crew member in any order permanently unlocks a distinct gameplay system while canon stays fixed; a first community meeting where the player helps set the station's first major priority among six equally valid options; a collaborative construction part where major projects require multiple crew members and present quick-versus-reinforced and immediate-versus-delayed-accessibility tradeoffs; and eight First Tests, one per primary crew member, that must all be completed before the chapter ends. The chapter closes with the unknown signal returning stronger, longer, and intentional, setting up Chapter Four. A section design framework was adopted alongside it and promoted to a global canon rule in the Canon Rules document, "Every Major Decision Answers Four Questions": every major player-decision section answers four questions, namely what decision the player makes, what gameplay system is learned, what philosophy or life lesson is learned, and what permanently changes as a result. Chapter Three is the first chapter authored to that rule; the chapter doc references it rather than restating it as its own governance.

### Chapter Four: Salvage designed from the Maintenance perspective

The full part-by-part design for Chapter Four, Salvage, was written as a design note across six parts. Chapter Four introduces the expedition gameplay loop that defines the remainder of Operation One: for the first time the player deliberately leaves home, not to escape Hearth but to return to it and learn from it, salvaging history as much as technology. Its primary question is whether the crew can build something worth protecting, answered yes by remembering what came before, and it ends by asking what really happened aboard Hearth before its destruction. The chapter teaches a repeatable expedition rhythm of preparing, departing, exploring, recovering, returning, reflecting, and rebuilding, through six parts: preparing for the first expedition with three pre-departure decisions; departure with route selection and advisor interaction; the first expedition with four explorable zones chosen in any order; homecoming with a community debrief and four salvage-integration decisions; living with discovery, covering return expeditions, artefact classification, and the expedition journal; and expedition consequences, completing the decision framework. The chapter climaxes when the strange signal from Chapter Three responds not to the station or the player but to something recovered from Hearth, and it closes asking what Hearth was carrying that someone still wants.

Alongside Chapter Four, two expedition rules were promoted to global canon rules in the Canon Rules document. "Every Major Expedition Begins and Ends at Home" establishes the home as the emotional, mechanical, and narrative anchor, fixes the seven-beat expedition rhythm, and requires every expedition to return with at least one meaningful contribution to the community rather than resources alone. "Every Expedition Expands Understanding" requires every expedition to answer one important question and create at least two new ones, so curiosity always outpaces certainty. The chapter's own recurring canon notes, that zones are chosen rather than completed, that no zone is ever fully completed on the first visit, and that completion should inspire curiosity, are given distinct headings within the chapter doc so they remain screen-reader navigable.

### Chapter Five: First Contact designed from the Maintenance perspective

The full part-by-part design for Chapter Five, First Contact, was written as a design note across six parts. Chapter Five is the player's first encounter with an unmistakably non-human civilisation: the player does not meet living Shawabti but their preserved legacy, and the chapter introduces archaeology, translation, environmental interpretation, and ethical exploration, teaching that exploration is stewardship rather than ownership. Its primary question is what the Shawabti left behind, answered as something intentionally preserved, and it ends by asking why it was preserved for us. The six parts are the signal, which resolves the Chapter Four mystery into an intentional, waiting transmission and sends the crew to a single perfectly intact structure; the threshold, where the structure opens not because a puzzle was solved but because the player observed before acting; the first archive, where collaborative observation projects the first living reconstruction of the Shawabti, seen but not understood; echoes, where the player walks through a preserved moment of Shawabti society as an observer who cannot change or collect anything; resonance, where the structure reveals it was evaluating how the player explored, offers a preservation-versus-recovery artefact decision with no morally correct answer, and ends with the damaged helmet quietly speaking a single Shawabti word; and learning resonance, which introduces Resonance as the central gameplay system of the franchise with permanent unlocks. The chapter is consistent with established canon that Faience is the player's AI companion tied to the damaged helmet whose heads-up display was lost during the escape, and it closes on the hook that Faience knows a language nobody has translated, asking who or what Faience is becoming, which becomes the foundation of Chapter Six. The chapter title First Contact was previously unlocked and is now set. The chapter's in-chapter canon notes, that no ending choice is morally correct, that every Resonance interaction supports multiple methods, and that no Resonance interaction requires one ability, are given distinct headings so they remain screen-reader navigable.

A design adjustment was applied to the accessibility reveal at the user's direction. The player should not learn in a single reveal that Shawabti civilisation was built around accessibility; that understanding is meant to unfold gradually across Chapters Five, Six, and Seven. In Chapter Five the player should simply notice that the structure seems to work for everyone without yet understanding why, preserving the mystery. Accordingly, the explicit accessibility-conclusion statements in the source material for Part Three and Part Six were softened to defer the "why" while keeping the principle that the game never directly explains this conclusion and that players arrive at it naturally through repeated experience.

### Expedition Success Evaluation adopted as a global canon rule

A new franchise-wide rule was promoted to the Canon Rules document, "Every Expedition Records How the Player Approached Discovery." Every major expedition records the player's approach to discovery as philosophy rather than morality, and never as a numerical score. Possible expedition philosophies include curiosity, preparation, preservation, collaboration, leadership, scientific investigation, engineering, compassion, historical preservation, and accessibility. These quietly influence crew dialogue, Knowledge Archive entries, Resonance responses, future advisor recommendations, environmental storytelling, optional missions, and historical interpretation. The player is never told they played correctly; the world simply remembers how they chose to build civilisation, so the player becomes known for how they think rather than becoming good or bad. Chapter Five documents this system in its final part, building directly on the resonance chamber's evaluation of the player's approach so that the structure's judgement becomes the first entry in a record that continues across every future expedition.

### Chapter Six: The Living Archive designed from the Maintenance perspective

The full part-by-part design for Chapter Six, The Living Archive, was written as a design note across six parts. Chapter Six transforms the Knowledge Archive from a library into an active participant: the player stops studying isolated discoveries and begins assembling an entire civilisation from fragments, learning that preserving knowledge also preserves values, relationships, and ways of thinking. Its primary question is what the Shawabti were trying to preserve, answered as more than knowledge, and it ends by asking why Faience understands them. For the first time the destination is not primarily physical; the player explores the Living Archive itself, which is established as a preserved civilisation communicating through Resonance rather than a database, a search engine, or artificial intelligence. The six parts are awakening, where the damaged helmet activates on its own and Faience says it remembers; conversations across time, which teaches that understanding grows through conversation and progressive translation rather than information retrieval; reconstructing a civilisation, which teaches historical synthesis by connecting relationships rather than isolated events and introduces the Investigation Board; humanity in the archive, which reveals the Shawabti deliberately observed early humanity without interfering; the keeper, which concludes the narrative arc by revealing the Archive was preserving a relationship and frames Faience as the first bridge between two civilisations; and carrying the archive forward, which makes the Living Archive a permanent part of the station and completes the chapter's systems. The chapter maintains established canon that Faience is the player's AI companion tied to the damaged helmet, and it deepens the gradual awakening arc so that Faience increasingly appears to remember the Shawabti rather than merely translate them. It closes on Faience's first original, untranslated question and a distant Shawabti structure beginning to transmit, and it poses the question that sets up the conclusion of Operation One: what kind of civilisation will humanity choose to become. The chapter title The Living Archive is now set.

Alongside Chapter Six, the chapter's own canon rule that knowledge must always produce action was reconciled into the existing global rule rather than added as a second, competing rule. Chapter Six documents the rule within its final part under the heading "Knowledge Must Always Produce Action" as an explanatory restatement, while the master Canon Rules document keeps a single authoritative rule, "Knowledge Must Always Change Something," which was expanded to state explicitly that knowledge must always produce action and to enumerate the fuller list of things a major discovery must eventually change, namely dialogue, exploration, engineering, medicine, leadership, diplomacy, accessibility, research, relationships, community life, or player understanding, with the same closing principle that if knowledge changes nothing it should not exist. This preserves the single-source-of-truth rule that the project forbids competing canon rules. The chapter's two concluding beats, the narrative climax in the fifth part and the systems-and-epilogue climax in the sixth part, are given distinct headings so the two climaxes and their end-state summaries remain unambiguous and screen-reader navigable, and only the final part closes the chapter.

### Chapter Seven: Resonance designed from the Maintenance perspective

The full part-by-part design for Chapter Seven, Resonance, was written as a design note across four parts and an epilogue. Chapter Seven is the conclusion of Operation One and introduces no new major gameplay systems; instead every system from Chapters One through Six is combined into increasingly complex, integrated situations so the player demonstrates mastery through application rather than instruction. Its primary question is what kind of civilisation humanity will become, its answer belongs to the player, and its ending marks the close of Operation One and the beginning of Operation Two. The invitation begins the finale as every discovered Shawabti structure activates at once as a single complete harmonic sequence, and the community chooses to answer because it is ready rather than because it must. The journey teaches integrated decision making, where no challenge is solvable by one mechanic alone and Faience guides by recognising Resonance rather than giving directions. The choice of civilisations is the philosophical climax: a waiting chamber asks what knowledge, power, and memory should become, evaluating the civilisation the player has built rather than the individual. The first human archive completes Operation One by turning the station into a permanent home, research centre, Living Archive, school, medical centre, engineering facility, cultural centre, and launching point, and it records humanity's first Archive entry through any communication method the player chooses, all treated as equally valid. The epilogue, A New Morning, hands the player an unrestricted living sandbox so they can experience the consequences of their choices before Operation Two.

The chapter carries three canon statements, each documented in-chapter under a distinct heading so the repeated label "Canon Rule" never appears ambiguously: the Shawabti preserve the player's philosophy without judging it, Operation One ends without final answers, and Operation One never truly ends because the station remains the player's home for the entire franchise. These are recorded as thematic conclusions restated from existing rules rather than promoted as new competing global rules, consistent with the single-source-of-truth principle. This entry also records that the revision request that prompted this work assumed the chapter already existed, but the target file was absent at that time, so Chapter Seven was created from the attached work order with the naming corrected from the start rather than edited after the fact. During this work the Vale System naming was corrected to canon: the morning announcements in both the final cinematic scene and the epilogue are delivered by Captain Elian of the Vale System using the protocol "This is Captain Elian of the Vale System," and the erroneous "Captain Mara Vale" appears nowhere; Mara Chen remains the blind engineer and is not a member of the Vale System. With Chapter Seven written, its title, Resonance, is now set, and every Operation One chapter title is locked.

### Operation One master timeline established as the canonical chronology

The full master timeline for Operation One: Exodus was written as a design note and is now the single canonical chronology beneath the Campaign Bible, the chapter design notes, and the Crew Progression System. It records the seven ordinary days aboard CSV Hearth and the attack that ended them, with each day written as a complete chronological narrative covering the night shift, the morning announcement, every working and living department, midday, evening, night, the state of the anomaly, the presence of each primary character, the soundscape, environmental storytelling, accessibility, cross-perspective anchor scenes, and the narrative purpose, gameplay purpose, emotional goal, and end state of the day. Attack Day is written minute by minute, tracking every department, every captain, and every primary character simultaneously, and it ends at the exact moment Chapter Two begins, with the player waking alone inside the damaged escape pod.

Two structural decisions were made to keep the timeline consistent with existing canon. First, the timeline counts down toward the attack, so that Day Seven is the earliest recorded day and Day One is the eve of the attack, while Chapter One's design note counts its playable scenes forward from the player's first morning. Rather than let the two documents contradict each other, the timeline includes an explicit reconciliation, stating that a Chapter One scene labelled forward day number N corresponds to the timeline's countdown day numbered eight minus N, which makes Chapter One's first morning the timeline's Day Seven; the morning announcement quoted in Chapter One is therefore preserved unchanged as the Day Seven announcement, and the six later announcements are new and build the themes of community, belonging, responsibility, hope, memory, and civilisation toward a Day One announcement written to become devastating in hindsight. Second, the work order referenced a Perspective Map and a Character Schedule that do not yet exist; the timeline anticipates both by name without linking to them and remains the authoritative chronology until they are written. Attack Day was also established as Founding Day, the colony's founding anniversary, so that the entire week's festival preparation points toward a celebration that never happens, and the interdisciplinary anomaly meeting scheduled for that morning never convenes. The timeline's six canon rules are documented within the timeline under distinct headings and are recorded here rather than duplicated into the master Canon Rules document, consistent with the single-source-of-truth principle, and Faience is kept absent for the entire week, its first faint stirring occurring only in the damaged pod on Attack Day, so that the established rule that Faience is not active before the attack is preserved.

The character schedule for Operation One: Exodus was written as a design note and is now the canonical reference for where every recurring character is at any point in the week aboard CSV Hearth and during the attack. Where the master timeline answers what happened, this document answers where everyone is when it happened, and it is authoritative over any other document that disagrees about a character's whereabouts or routine. It gives every recurring character a complete identity, a full daily routine, professional responsibilities, community relationships, accessibility profile, sound identity, cross-perspective presence across all eight profession campaigns, and an emergency routine whose Attack-Day position matches the master timeline's minute-by-minute account and its separate-pod scatter. It documents its own canon rules in-document rather than adding competing entries to the master Canon Rules document, consistent with the single-source-of-truth principle. Faience is handled honestly as an absence for the entire ordinary week, with only a first faint stirring in the damaged pod on Attack Day, preserving the established rule that Faience is not active before the attack. This document was written before the referenced Perspective Map exists; the Perspective Map is named as a forthcoming companion without being linked or invented.

### Modular architecture bible established as the franchise architectural source of truth

The modular architecture bible was written as a design note and is now the architectural source of truth for the Resonance franchise. It defines every reusable room, space, and facility module from which locations are assembled, and it is explicitly not specific to CSV Hearth; Hearth is built from these modules, and so is every future station, colony, ship, settlement, embassy, research outpost, and military installation. Its canon rule is that a room exists independently of the ship containing it, so only its location changes while its purpose stays consistent, which makes a player's mastery of any module transferable to every location that uses it and is therefore also an accessibility guarantee. Sixty-six modules are defined, each written to an identical five-part structure covering overview and purpose, accessibility and universal use, soundscape and lighting and environment, equipment and connections, and gameplay and narrative and emergencies, with every field fully fleshed out and every heading disambiguated by module name. Each module specifies how it sounds, how blind players navigate it, how Deaf players receive information, how wheelchair users move through it, and how children, civilians, and professionals use it, along with how it behaves in emergencies and after upgrades. Modules connect through standardised connection points so that entire locations can be assembled from the same architectural language rather than rebuilt from scratch. Module content is kept consistent with the crew and the master timeline, and the Escape Pod Bay is kept consistent with the Attack-Day scatter in which each primary crew member escapes in a separate pod. Because this document was large, it was produced by writing the front matter and end matter directly and generating the module sections in parallel fragments that were then assembled and validated as a single living document. The two documents in this entry were commissioned as work orders numbered 009 and 010 by the requester; because the master timeline was work order 008 and the requester asked that the numbering advance, the character schedule is recorded as work order 010 and the modular architecture bible as work order 011.

### Perspective Map for Operation One: Exodus

The Perspective Map for Operation One: Exodus was written as a design note and is now the canonical reference for how each of the eight playable professions experiences the identical events and spaces of the operation. Where the master timeline answers what happened and when, and the character schedule answers who is where, this map answers how each profession lives the same history, and it is authoritative over perspective while never moving an event or a person that the other two documents fix. The eight professions are the ones defined in the Campaign Bible: medic, scientist, engineer, maintenance, security, commissioned officer, non-commissioned officer, and civilian, and each is written against the eight perspective dimensions the Campaign Bible establishes. The map documents its own canon rules in-document rather than adding competing entries to the master Canon Rules document, consistent with the single-source-of-truth principle. Three canon points were fixed in the writing. The player is a member of the chosen profession rather than that profession's named department lead, which keeps the named recurring crew, Mara Chen, Doctor Samira Kade, Security Lead Talia Rusk, Dr. Soren Qadir, Chief Petty Officer Jonah Briggs, Rowan Ives, Mira Ives, and the Vale System, as the crew the player rescues one by one in Chapter Two rather than as the player's own character. The separate-pod scatter recorded in the master timeline is unchanged, with each named lead launching from the pod the timeline fixes for them. Faience remains absent from every profession's ordinary week, and whatever profession the player chooses, the player ends Attack Day in a separate damaged pod where Faience first stirs, the same seam Chapter Two opens inside. With the map written, the three documents that previously named it as forthcoming, the master timeline, the character schedule, and the modular architecture bible, were updated to reference the existing document, and the design-notes index now lists it. The maintenance perspective, already written in seven chapters, is treated as the reference the other seven professions will match when their full campaigns are written.


  ### House-style heading disambiguation for the older design notes

  The crew progression system and the maintenance chapter notes two through seven were brought into line with the house style rule that no two headings in a document may be identical, so that every section is uniquely reachable by a screen reader. These documents had repeated the same generic subsection headings, such as Role, Purpose, Accessibility, Sound Design, and Emotional Goal, under many parent sections, which produced dozens of colliding headings per file. Each duplicated third-level heading was prefixed with the name of its parent section, using the same Name then subsection form already used by the character schedule and the perspective map, so that a heading like Purpose under Part Two becomes Part Two then Purpose, and a heading like Role under Rowan Ives becomes Rowan Ives then Role. The change was confined strictly to heading lines. No body text, canon, ordering, or meaning was altered, and headings that were already unique were left untouched to keep the change minimal. Chapter one already satisfied the rule and was not modified. After the pass, no document in the repository contains a duplicate heading.

### The Vale System and fronting promoted to canon rules

Two canon rules about the Vale System were promoted into the master Canon Rules document so that the whole set can be seen in one place. The first rule, that the Vale System is one shared body, states plainly that the Vale System is one person inhabited by four members who front in turn, that those members are Captain Elian, Captain Naomi, Captain Gideon, and Captain Iris of the Vale System, and that the system is never written as four separate people and never as the erroneous Captain Mara Vale. It records which things stay constant across every member, namely the shared body, the captain's chair and its authority, the history the crew has with the system, and the care each member holds for the ship, and which things change as one member fronts and then another, namely voice and cadence, manner and priorities, the situations each member meets, and the name the member answers to, and it repeats that Mara Chen is the blind engineer and is not a member of the Vale System. The second rule, that fronting is gameplay, states that which member is fronting is part of the game rather than decoration, that the fronting member aligns to the situation with Elian on ordinary mornings, Naomi in emergencies, Gideon on failing systems, and Iris with civilians, that a switch is rarely announced, and that the player recognises the current member through observation rather than a label, which makes reading the Vale System a skill the player builds. These rules do not compete with the existing Vale System material in the Crew Progression System and the Character Schedule; they consolidate it, and the Crew Progression System's note that the player recognises each member by voice was widened to say the player recognises each member by behaviour, reading voice, manner, and priorities, so that no wording drifts from the promoted rules. The Relationship to Other Documents section of the Canon Rules document was updated to point to the Crew Progression System and the Character Schedule as the places these rules are elaborated.

### Gameplay Progression Matrix established as the franchise progression reference

The Gameplay Progression Matrix was written as a design note and is now the single reference that gathers every form of progression in Resonance into one place. It creates no new canon; it consolidates progression that already lives across the Bibles, the Crew Progression System, and the chapter notes, and where any wording drifts, the source documents govern. The matrix opens with the progression philosophy that Resonance never advances the player through arbitrary levelling, that every form of progression is something the player can hear, feel, or do rather than a stat, and that people are the primary progression system while knowledge, accessibility, sound, home, Faience, perspective, and expeditions all support them. It then defines each progression system against an identical set of attributes, follows those systems chapter by chapter through a campaign matrix, and closes with the end state of Operation One and the reminder that the campaign is complete only when every profession has been built on the same timeline. The design-notes index was updated to include the matrix, and every heading is disambiguated by its parent so the reference is navigable by a screen reader.

### Character, cast, and representation canon rules adopted

A batch of canon rules governing characters, the player's own character, the Vale System, and cast representation was promoted into the master Canon Rules document so that the whole set can be seen in one place, consistent with the established practice of keeping the Canon Rules document as the single global list beneath the Project Constitution. Several new rules were added. ### World persistence, player residency, and safe logout adopted

A set of canon rules was adopted defining how the world behaves around an absent player and how players relate to a persistent, multiplayer world. The world never pauses when a player disconnects or logs out; the world, not the player, hosts the game; the persistent entities are stations, settlements, expeditions, communities, research missions, and recovery efforts, which continue when any member leaves; players are residents who join an existing civilisation rather than opening a lobby or creating one; and every persistent world holds its own history, infrastructure, community Knowledge Archive, rescued crew, projects, accessibility improvements, engineering upgrades, memorials, and culture, to which players contribute without owning it, consistent with the rule that the player never alters canon. Logout handling was defined through Protected Personal Spaces, where an absent character is safely off duty and is never harmed, moved, or reassigned; an unsafe-logout priority order that moves a character toward safety; a safe-return summary of meaningful events; community redistribution of responsibilities so that no player is punished for logging out; and an emergency exception that withdraws the character from danger and limits any offline participation to realistic defensive or evacuation behaviour so that a player's absence never causes catastrophic community failure. The rules were promoted into the Canon Rules document as the master statements, with the two identical statements that the world continues merged into a single rule, and the same material was elaborated in the Network and Multiplayer Bible. No canon conflicts were found; the rules align with the existing multiplayer, accessibility, and player-agency canon.
